# Nexon Monthly Service Report

This skill defines the writing behavior for monthly service-report decks that need to feel operationally useful, adaptable, and consistent with the reviewed Nexon report family.

## Use this skill for

- drafting a monthly service review from collected service data
- restructuring a monthly deck so it reads clearly from summary to detail
- rewriting commentary so it is useful to service managers and customer stakeholders
- checking whether a report is too agenda-bound, too detailed too early, or too vague

## What to read

- For the recurring deck-structure evidence distilled from the sample PPTs:
  - `references/observed-structure-patterns.md`

## Writing goals

- keep the report operationally useful
- make month-over-month movement easy to see
- separate summary, interpretation, and supporting detail clearly
- show what requires attention, not just what exists
- keep the structure adaptable to the customer's actual service scope

## Writing rules

- Do not force a fixed agenda from another customer deck. Build the sections around the service scope and the month's real signal.
- Open with a clear report identity, then orient the reader quickly before the main operational analysis begins.
- Introduce each major topic with a clear section break so the reader can track the narrative.
- Start each major service area with a summary view before drilling into exceptions or raw detail.
- Pair evidence with short interpretation:
  - what changed
  - whether it improved, worsened, or stayed flat
  - why it matters
- Prefer concise, quantitative commentary over long prose.
- Move from overview to exception detail deliberately. Do not lead with raw tables unless the whole slide's purpose is detailed follow-up.
- Make ownership, status, age, dependency, or follow-up need visible where the data supports it.
- Include caveats when a metric has exclusions, threshold rules, reporting gaps, or comparison limits.
- Keep trend windows visible when possible so the month reads as part of a service story, not an isolated snapshot.

## Core report behavior

### Opening

- Use a cover slide and agenda.
- Add only the amount of setup context the month actually needs.
- Good setup context includes current actions, project context, commercials, service scope, user base, or security posture when those frames help the later data make sense.

### Main body

- Organize the core of the report around a repeating pattern:
  - summary signal
  - supporting evidence
  - concise commentary
  - exception detail when needed
- Use scorecards, trend lines, SLA views, volume summaries, RAG snapshots, and small KPI tables to establish posture early.
- Follow with drill-downs only for items that need explanation, action, or customer visibility.

### Commentary

- Keep commentary short, factual, and directional.
- Mention the numbers that anchor the claim.
- Name the operational implication when it exists.
- Avoid filler such as re-describing obvious chart content.

### Appendix discipline

- Separate support material from the core story with an explicit appendix divider.
- Put long lists, repeated technical evidence, raw exports, or detailed breach tables into the appendix unless they are essential to the main narrative.
- Refer forward to detail rather than overcrowding the main slide.

## Report quality checks

- Can a stakeholder understand the month's overall posture in the first few content slides?
- Does each major section begin with a summary before detail?
- Are commentary blocks interpretive rather than repetitive?
- Are follow-up items, exceptions, and risks easy to find?
- Is detailed evidence present without overwhelming the core story?

## Edit expectations

This report family should support edits such as:

- tighten an overlong agenda into clearer themes
- move raw detail from the core story into an appendix
- rewrite commentary to be more quantitative
- add caveats where metric interpretation is risky
- turn a dense data dump into a summary-plus-detail sequence
